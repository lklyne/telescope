# Canvas drag affordances — axis lock + alignment & distribution guides

Source: https://github.com/lklyne/specular/issues/91

> *This was generated by AI during planning (grill-with-docs → to-prd).*

## Problem Statement

Designers and developers using Specular's canvas don't have the basic spatial-arrangement affordances they expect from FigJam and Figma. Specifically:

1. There's no way to drag an entity while holding `Shift` to constrain its movement to a horizontal or vertical axis. Aligning entities by eye while preserving the other axis is tedious — users have to drag, undo, and retry, or use the inspector to manually zero one coordinate.
2. There are no visual snapping guides that confirm when a dragged or resized entity aligns with another entity's edge or center. Users have to zoom in to read coordinates or eyeball it, then second-guess the result.

The 20 px grid-snap helps with rough placement but doesn't tell the user *when* their entity is aligned with a neighbor, doesn't help with cross-axis center-alignment between two stickies, and doesn't survive any entity placed off-grid.

## Solution

Add two FigJam-parity drag affordances. Both terms are defined under `Drag affordances` in `CONTEXT.md`; the architectural trade-off is captured in [ADR 0012](../../docs/adr/0012-alignment-guides-are-visual-only.md).

1. **Axis lock.** Holding `Shift` during an entity drag (with or without `Alt`/`Option` for copy) constrains the movement to a single axis. The lock is **live** — the dominant axis is recomputed every frame from the cursor's offset to the drag origin, so the user can press, release, or flip H↔V mid-drag without releasing the mouse. The locked axis bypasses grid-snap so the entity holds its exact origin coordinate.

2. **Alignment & distribution guides.** During an entity drag or resize, Specular paints 1 px solid lines in the canvas accent color whenever the dragged (or resized) entity's edge or center coincides with another entity's edge or center within 0.5 px (post-grid-snap). Each candidate contributes 6 edges (top / bottom / left / right / horizontal-center / vertical-center); the dragged entity contributes its own 6 reference points (only the moving edge for resize). When the dragged entity sits at equal distance from two or more candidates along an axis, `==` measure marks render as **distribution guides**. The candidate set is a snapshot taken at gesture-begin of all entities whose rect intersects the canvas viewport, excluding the active selection. Pages and groups (snap to bbox) are included.

Guides are **visual-only**: grid-snap remains the only magnetic pull, and guides only render when the post-snap position is genuinely aligned. They confirm alignments — they never create them. See ADR 0012 for the rationale and pivot path.

## User Stories

1. As a Specular user, I want to hold `Shift` while dragging an entity so that the entity is constrained to horizontal or vertical movement only, so that I can move things without disturbing the other axis.
2. As a Specular user, I want the axis lock to choose the dominant axis from my cursor's offset to the drag origin, so that the constraint follows my intent rather than the first jitter of my hand.
3. As a Specular user, I want to press and release `Shift` mid-drag, so that I can engage and disengage axis lock without releasing the mouse.
4. As a Specular user, I want to flip H↔V mid-drag by arcing my cursor past 45° while `Shift` is held, so that I can correct mistakes without restarting the gesture.
5. As a Specular user, I want axis lock to bypass grid-snap on the locked axis, so that an entity returns to exactly the coordinate it started at when I release the mouse on that axis.
6. As a Specular user, I want axis lock to still grid-snap on the free axis, so that the placement feel stays consistent with normal drag.
7. As a Specular user dragging a multi-selection, I want the constrained delta applied uniformly to every selected entity, so that the group moves as a rigid unit along one axis.
8. As a Specular user, I want `Shift` + `Alt`/`Option` to compose, so that I can axis-lock a copy-drag while duplicating.
9. As a Specular user, I want alignment guides to appear when my dragged entity's top, bottom, left, right, horizontal-center, or vertical-center coincides with another entity's matching reference, so that I get instant visual confirmation of alignment.
10. As a Specular user, I want guides to only appear when the alignment is genuinely true (within 0.5 px after grid-snap), so that I can trust what I see and not be misled by guides that lie.
11. As a Specular user, I want the guide line to span only the rects involved (from min-edge to max-edge of the candidate-and-dragged pair on the snapping axis), so that the screen isn't cluttered with full-viewport crosshairs.
12. As a Specular user, I want guides to disappear the instant alignment is no longer true, so that I have a clean signal of "you're aligned now" / "you're not."
13. As a Specular user, I want guides to consider only viewport-visible entities, so that the system isn't spending budget on entities I can't see and the guide list stays focused on what's relevant on my screen.
14. As a Specular user, I want pages (live web items) treated as snap candidates like any other entity, so that I can align stickies and shapes to a page's edges or center.
15. As a Specular user, I want groups treated as snap candidates by their outer bounding rect (not by every child individually), so that group alignment behaves intuitively.
16. As a Specular user, I want the active selection excluded from snap candidates, so that a dragged entity never tries to align with itself.
17. As a Specular user, I want the snap candidate set frozen at the moment I begin the drag (pan is locked during drag anyway), so that the guide behavior is stable and predictable for the duration of the gesture.
18. As a Specular user, I want `==` distribution guides to appear when my dragged entity sits at equal distance from two or more other entities along an axis, so that I can build evenly-spaced rows and columns visually.
19. As a Specular user, I want alignment and distribution guides to fire during resize as well as drag, so that I can stretch an entity precisely to match another's edge or center.
20. As a Specular user resizing an entity, I want only the moving edge to contribute dragged reference points, so that the opposite (anchored) edge doesn't spuriously trigger guides.
21. As a Specular user, I want guides to render above entity bodies and selection chrome, so that they aren't occluded by what I'm dragging.
22. As a Specular user, I want axis lock and guides to compose with `Alt`/`Option` copy-drag, so that copies snap and constrain too.
23. As a Specular user, I want the grid-snap behavior I'm used to (20 px) to remain unchanged, so that nothing about my existing placement habits regresses.
24. As an agent / CLI / smoke test driving the canvas, I want axis lock and guide detection to be expressible through the same `applyDragDelta` path that interactive drags use, so that automation produces identical results to a human drag.
25. As a Specular maintainer, I want the axis-lock projection, snap candidate snapshot, alignment detection, and distribution detection to be pure functions tested in isolation, so that regressions in geometry are caught fast and without spinning up Electron.

## Implementation Decisions

**Deep, isolated modules (the four pure ones tested in `tests/unit/`):**

- `axisLockProjector` — `(rawDelta, cursorOffsetFromOrigin, shiftKey) → constrainedDelta`. Picks dominant axis from cursor offset and zeros the smaller component. No-op when `shiftKey === false`. Pure; no Electron.
- `snapCandidateSnapshot` — `(workspaceState, viewportRect, excludedIds) → SnapCandidate[]`. Returns each viewport-visible entity as `{ id, kind, top, bottom, left, right, hCenter, vCenter }`. Pages included; groups represented as their bbox (children inside an included group do not double-contribute); selection excluded. Captured once at gesture-begin.
- `alignmentGuideDetector` — `(draggedRects, candidates, tolerance) → AlignmentGuide[]`. For each axis, finds dragged-ref ↔ candidate-edge pairs within 0.5 px. Emits guides carrying axis, target coordinate, the rect-span endpoints needed to paint the line, and the candidate entity id.
- `distributionGuideDetector` — `(draggedRect, candidates, axis, tolerance) → DistributionGuide[]`. Detects when the dragged rect's gap to one candidate equals its gap to another (along the axis) within tolerance. Emits chains with the gap value and the involved rect endpoints.

**Pipeline seam (modified):**

- The existing `applyDragDelta` (the one site in main that turns a raw drag delta into entity position changes) gains two stages: axis-lock projection runs *before* `snapToGrid`; alignment + distribution detection runs *after*. Single seam where the four pure modules meet; nothing else in the runtime needs to know about guides.

**Resize adapter (new):**

- Resize gestures are routed through an adapter that translates the handle-drag delta into "dragged ref points" — only the moving edge(s) contribute references. The adapter consumes the same `snapCandidateSnapshot` captured at resize-begin and feeds the alignment detector. No new IPC vocabulary; the existing resize IPC just gains the same guide-broadcast pattern.

**IPC payload changes:**

- `canvas-drag-entity` update payload gains a `shiftKey: boolean` field. The renderer reads `event.shiftKey` and forwards; main owns projection.
- A new `canvas-guides` broadcast from main carries the per-update guide list (alignment + distribution). Sent on every drag/resize update; empty list clears the overlay. Sent on commit/cancel to clear.

**Renderer (new layer in aboveView):**

- `GuideOverlayLayer` is a sibling of the existing `DragCopyPreviewLayer` in `aboveView`. It subscribes to `canvas-guides` and paints 1 px solid lines for alignment guides (spanning the involved rects) and `==` marks for distribution guides. Pure presentation — no business logic, no state of its own beyond the subscription.

**Architectural respect:**

- All projection and detection happens in main (`src/main/runtime/`), per the interaction-layer invariant that main owns workspace state and renderers are clients. No second projection lives in the renderer (rejected during grilling to keep the snap engine consolidated).
- No new `InteractionMode` is introduced. Axis lock and guides are payload features of the existing `dragging-entities` and `resizing-entity` modes, per the spec's "5.6 Should I add a new interaction mode?" heuristic.

**Out of scope deviations from FigJam:**

- Center alignment will rarely engage because the post-grid-snap position is unlikely to satisfy the 0.5 px equality on centers (an entity's center sits at `x + width / 2`; off-grid unless width is a multiple of 40). This is the explicit trade-off captured in ADR 0012 — the cost of guides being honest given the 20 px grid stays.

## Testing Decisions

**Good-test bar:** Each test asserts an external behavior (function input → output), not internals. A test "passes its bar" if it would have caught a real regression (you can name the production-code mutation that breaks it), survives a refactor, and is locally legible from the test file alone. This matches the project standard in `tests/README.md` (per issue #81).

**Unit tests (all four pure modules), in `tests/unit/`:**

- `tests/unit/axis-lock-projector.test.ts` — dominant-axis selection across the 8 octants of cursor offset; mid-drag flip behavior as cursor crosses 45°; no-shift passthrough; ties-at-exact-45° resolved deterministically; multi-select implication (same delta returned regardless of count, since callers apply it uniformly).
- `tests/unit/snap-candidate-snapshot.test.ts` — viewport intersection (entity fully inside, partially clipping, fully outside); selection exclusion; page kind included with its full bbox; group kind contributes its bbox once, children inside the group don't double-contribute; empty workspace → empty snapshot.
- `tests/unit/alignment-guide-detector.test.ts` — 6×6 cross of dragged-ref × candidate-edge pairs across both axes; 0.5 px tolerance boundary (inside and outside); no-match returns empty; multi-candidate match (nearest wins per axis); span endpoints span both rects correctly.
- `tests/unit/distribution-guide-detector.test.ts` — equal-gap triples; near-miss at tolerance boundary; multi-candidate chains (four entities forming three equal gaps); mixed axis (horizontal distribution doesn't fire on vertical-only patterns).

**Smoke test (one), in `tests/smoke/`:**

- `tests/smoke/drag-affordances.test.ts` — exercises the integrated path end-to-end via the existing `AppClient` HTTP API: place two stickies, drag one with `Shift` held, assert the entity's `canvasX` (or `Y`) is unchanged and the moved axis grid-snaps; place two stickies vertically aligned, drag one near alignment, assert a `canvas-guides` broadcast emits the expected guide structure; commit and assert the guide list clears.

**Prior art:**

- Unit-test style for pure runtime helpers: `tests/unit/zoom.test.ts` (behavior-driven blocks, per the post-issue-#81 standard), `tests/unit/interaction-controller.test.ts` (public-API-driven, observable outputs).
- Smoke harness: `tests/smoke/test-utils.ts` `AppClient`; comparable file shape `tests/smoke/gestures.test.ts`.
- IPC payload extension reference: how `canvas-drag-copy-selection` flows through `register-canvas-drag-ipc.ts` is the closest analog for the `shiftKey` payload addition and the `canvas-guides` broadcast.

## Out of Scope

- **Magnetic alignment guides.** Guides are visual-only in v1 (ADR 0012). Making them pull is a separate, larger product decision that probably implies dropping or shrinking grid-snap.
- **Canvas-center guide.** FigJam shows a center-of-canvas alignment line for the page itself; deliberately deferred.
- **Suppressing-snap modifier.** No second modifier (e.g. `Cmd`/`Ctrl`) to disable grid-snap or guides; `Shift` is claimed for axis lock.
- **Smart spacing manipulation.** Distribution guides are detection-only — they don't add the FigJam interaction where you drag a distribution gap to redistribute.
- **Edge-drag alignment.** Connection-edge drags don't snap to entity edges.
- **Marquee alignment.** Marquee gestures don't trigger guides.
- **Cross-canvas alignment.** Snap candidates come from the current canvas only.
- **Configurable tolerance / colors.** Both are constants in v1; no user-facing setting.
- **Performance regression coverage.** The snapshot-at-begin design is cheap by construction; no perf budgets are encoded in tests.
